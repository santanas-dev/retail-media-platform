## 1. Бизнес-сценарии

### Сценарий 1: Вход и безопасность

**Цель:** Аутентифицированный доступ к порталу через локальную учётную запись с полным RBAC/RLS.

**Описание:**
1. Пользователь открывает портал — перенаправляется на `/login`, если сессия отсутствует.
2. Вводит логин и пароль локальной учётной записи (bcrypt-хэши, httpOnly cookies).
3. После успешного входа — главный экран `/dashboard` с платформенной сводкой.
4. Все последующие страницы проверяют серверную сессию (signed cookies, server-side session store).
5. Для каждой страницы применяется проверка роли (`require_auth_for_page()`) и разрешений (`require_permission()` на каждом внутреннем endpoint).
6. Без сессии — редирект на `/login` с сообщением «Войти».
7. При выходе — очистка сессии, редирект на `/login`.

**Ролевая модель:** 8 ролей (system_admin, security_admin, ad_manager, approver, analyst, advertiser, operations, device_service), 47 разрешений.

**Безопасность:** JWT-аутентификация (60 минут), server-side sessions, RLS (Row-Level Security) — 7 типов scope, query-level enforcement. Нет JS/CDN/localStorage. Никаких raw UUID, secrets, tokens или backend URL в HTML.

**Критерий приёмки:** Пользователь успешно входит, видит главный экран, не может получить доступ к страницам без соответствующей роли. При отсутствии сессии — редирект на `/login`.

### Сценарий 2: Жизненный цикл креатива (загрузка → модерация → одобрение/отклонение)

**Цель:** Полный цикл: загрузка рекламного материала, автоматическая валидация, ручная модерация с maker-checker, принятие решения.

**Описание:**
1. Пользователь переходит на `/creatives`, нажимает «+ Загрузить креатив».
2. Выбирает файл (PNG/JPEG/MP4/WebM/GIF), указывает код, название, категорию.
3. **Автоматическая валидация при загрузке:**
   - Проверка MIME-типа: только разрешённые форматы (jpg, jpeg, png, gif, mp4, webm). **Загрузка `.mov` пользователем запрещена** — `.mov` не входит в `ALLOWED_UPLOAD_MIME_TYPES`.
   - Блокировка опасных типов: HTML, JS, SVG, ZIP, EXE, DLL, SH, PY и ещё 8 типов — отклоняются до MIME-проверки.
   - Проверка разрешения: **активный профиль 768×1024 portrait**. Креативы проверяются на соответствие этому профилю.
   - Проверка размера файла: ≤50MB (изображения), ≤100MB (видео), ≤20MB (GIF).
   - SHA-256: вычисляется инкрементально, дубликаты отклоняются (409 Conflict).
   - Для видео (MP4/WebM): проверка контейнера, кодека (h264/vp8/vp9/av1), длительности (≤30с), FPS (≤30), аудио запрещено.
   - Для GIF: проверка сигнатуры, кадров (≤300), длительности (≤15с), размеров.
4. После загрузки креатив получает статус `draft` (Черновик).
5. Пользователь отправляет креатив на проверку: статус → `pending_review` (Ожидает проверки).
6. Модератор (с ролью `approver`) переходит в очередь модерации `/creatives/moderation/queue`.
7. Берёт креатив в работу: статус → `in_review` (На проверке).
8. Проверяет: предпросмотр, метаданные, соответствие брендбуку, содержание.
9. **Принимает решение:**
   - **Одобрить** → статус `approved` (Одобрен). Действие записывается в аудит с флагом `av_warning` (если AV не настроен).
   - **Отклонить** → статус `rejected` (Отклонён) с комментарием с причиной.
   - **Вернуть на доработку** → статус `draft` с комментарием.
10. **Maker-checker:** Создатель креатива не может сам одобрить свой креатив — только другой сотрудник с правом `media.approve`. При попытке самосогласования: ошибка 400 + «Нельзя согласовать собственный креатив».
11. **Аудит:** Каждое действие модерации обязательно записывается в аудит (`creative.submit_review`, `creative.approve`, `creative.reject`, `creative.return_for_rework`, `creative.archive`). **Журнал аудита обязателен для каждого действия модерации.**

**Критерий приёмки:** Креатив успешно загружается только в разрешённых форматах. `.mov` загрузка пользователем отклоняется. Maker-checker срабатывает. Статус меняется по workflow. Аудит фиксирует каждое действие.

### Сценарий 3: Кампания + привязка креатива (только одобренный креатив)

**Цель:** Создание рекламной кампании и привязка к ней только одобренных креативов.

**Описание:**
1. Пользователь переходит на `/campaigns/create`.
2. Заполняет форму: код кампании (`campaign_code`), название, даты начала/окончания.
3. Нажимает «Создать» — кампания в статусе `draft`.
4. На странице кампании (`/campaigns/{code}`) нажимает «+ Креатив».
5. **Gate привязки:** система проверяет, что креатив имеет статус `approved`. Только одобренный креатив может быть привязан к кампании:
   - Креативы в статусах `draft`, `pending_review`, `in_review`, `rejected`, `validation_failed`, `archived` — не могут быть привязаны.
   - При попытке привязать неодобренный креатив — ошибка с бизнес-формулировкой.
6. После привязки креатив отображается в списке материалов кампании.
7. Статус кампании меняется по жизненному циклу: `draft` → `pending_approval` → `approved` → `published`.

**Критерий приёмки:** Кампания создаётся, креатив привязывается только в статусе `approved`. Неодобренные креативы не могут быть привязаны.

### Сценарий 4: Расписание и занятость эфира

**Цель:** Создание расписания показов, настройка слотов и проверка занятости рекламного времени.

**Описание:**
1. Пользователь переходит на `/schedule`.
2. Заполняет форму создания расписания: код, название, даты действия, часовой пояс.
3. Добавляет слоты: день недели, время начала/окончания.
4. Система отображает таблицу слотов.
5. На странице `/inventory` (Рекламное время) проверяется занятость эфира:
   - Форма с `device_code` и диапазоном дат.
   - Индикатор занятости: `occupancy_pct`, флаг `sold_out`.
   - Причины на бизнес-языке: «Занято кампанией X», «Служебное бронирование», «Аварийное».
   - Прогноз доступного времени (оценка по расписанию, confidence: low).
6. Конфликты расписания отображаются на `/reports` и в карточке расписания.

**Критерий приёмки:** Расписание создано, слоты настроены. Занятость эфира показывает реальные цифры на основе существующих кампаний.

### Сценарий 5: Процесс согласования (maker-checker)

**Цель:** Двухстороннее согласование — запрос на утверждение не может быть выполнен тем же лицом.

**Описание:**
1. На странице кампании пользователь нажимает «→ Запросить согласование».
2. Кампания переходит в статус `pending_approval`.
3. Заявка появляется на странице `/approvals` в виде карточки с деталями кампании.
4. **Maker-checker (обязателен):**
   - Инициатор запроса не может одобрить или отклонить собственный запрос.
   - Система проверяет: `requester_id != approver_id`.
   - При попытке самосогласования: ошибка 400 + бизнес-сообщение «Нельзя согласовать собственный запрос».
5. Другой сотрудник с ролью `approver` принимает решение:
   - **Одобрить** — кампания переходит в `approved`, запись в аудит.
   - **Отклонить** — кампания возвращается в `draft` с комментарием причины, запись в аудит.
6. **Журнал аудита обязателен** для каждого действия согласования.

**Критерий приёмки:** Maker-checker срабатывает. Собственный запрос не может быть согласован. Аудит фиксирует все решения.

### Сценарий 6: Подготовка публикации (физическая доставка заблокирована)

**Цель:** Формирование пакета публикации, генерация пакета рекламных материалов (manifest), backend-публикация. Физическая доставка на КСО остаётся заблокированной.

**Описание:**
1. После одобрения кампании пользователь нажимает «📦 Подготовить публикацию» на странице кампании.
2. На странице `/publications` отображается конвейер жизненного цикла пакета (batch lifecycle):
   - `draft` → `pending` → `approved` → `manifest_generated` → `published`.
3. Пользователь нажимает «→ На согласование» — пакет переходит в `pending`.
4. После одобрения пакета нажимает «📋 Сгенерировать пакет материалов» — система генерирует JSON-пакет рекламных материалов (manifest).
5. Статус пакета: «Пакет материалов готов».
6. Нажатие «🚀 Опубликовать (система)» — **backend-only публикация.**
   - Система выполняет публикацию только на уровне системы.
   - **Физическая доставка на КСО заблокирована.**
   - Отображается красный баннер NO-GO: «Физическая доставка на КСО заблокирована. Требуется закрытие 5 P0 блокировок.»
7. **Физический пилот остаётся заблокированным (5 P0 блокировок):**
   - B-01: Аппаратный сканер E2E не выполнен
   - B-02: Длительная проверка стабильности (48ч+) не выполнена
   - B-03: Физическая доставка пакета материалов на КСО не утверждена
   - B-04: Синхронизация агента на КСО не запущена
   - B-06: Развёртывание парка устройств не утверждено

**Критерий приёмки:** Пакет публикации создан, пакет материалов сгенерирован, backend-публикация выполнена. Красный баннер физической блокировки отображается.

### Сценарий 7: Отчёты и CSV-экспорт

**Цель:** Получение плановой отчётности в формате CSV с безопасной проекцией данных.

**Описание:**
1. Пользователь переходит на `/reports`.
2. Видит сводку распределения: «Кампании по статусам» (distribution bars).
3. Проверяет занятость эфира (прогресс-бар с порогами).
4. Просматривает конфликты расписания.
5. **CSV-экспорт (4 типа):**
   - `📥 campaigns_export.csv` — список кампаний с безопасной проекцией (без UUID, secrets).
   - `📥 airtime_export.csv` — занятость эфира по дням/слотам.

7. **Безопасная проекция:** никаких внутренних идентификаторов (UUID), технических полей, backend-путей в экспорте.
8. Фактические показы (Proof of Play) появятся после физического PoP flow — явно указано в интерфейсе.

**Критерий приёмки:** Все 4 CSV-файла экспортируются. Данные соответствуют RLS. Безопасная проекция подтверждена.

## 5. Политика безопасности (AV Security Posture)

### Текущий режим: `pilot_dev`

| Параметр | Значение |
|---|---|
| Режим AV-политики | `pilot_dev` |
| Требование AV-чистоты для публикации | `false` (`require_av_clean=false`) |
| Ручная модерация | ✅ Разрешена |
| Maker-checker | ✅ Обязателен |
| Журнал аудита | ✅ Обязателен для каждого действия модерации |
| AV-сканер | Не настроен (`scan_status=not_configured`) |

### Сообщение в интерфейсе

> «Проверка безопасности файлов ещё не настроена. Промышленный режим включать нельзя.»

### Ключевые правила

- **Имитация проверки безопасности (fake AV pass) запрещена.** Статус `scan_status=clean` никогда не выставляется автоматически без реального сканера.
- **Загрузка `.mov` пользователем запрещена.** Формат не входит в `ALLOWED_UPLOAD_MIME_TYPES`.
- **Активный профиль экрана: 768×1024 portrait.** Все креативы валидируются под этот профиль.
- **1440×1080 остаётся будущим/отложенным.** Поддержка полного Full HD с ad zone 1440×1080 отложена до этапа production на физической КСО.
- **Режим `pilot_dev` разрешает ручную модерацию.** Модератор может одобрить креатив без AV-сканирования с записью в аудит.
- **Производственный AV требует отдельного решения.** Включение режима `production` с требованием `scan_status=clean` — отдельный этап со своим approval gate.
- **В производственном режиме публикация без `scan_status=clean` должна блокировать одобрение и публикацию.** Система имеет контракт и guard для этого сценария.
