## RC0: Статус

**RC0 (Release Candidate 0)** — версия системы, готовая к backend-only бизнес-демонстрации. Все основные бизнес-сценарии проходят через портал без физической КСО. Физическая доставка, фактические показы и пилотный запуск остаются заблокированными до закрытия физических P0 gates.

---

## Что включает RC0

### Креативы и медиатека

- Загрузка креативов: PNG, JPEG, MP4, WebM, GIF
- Автоматическая валидация: MIME, размер, разрешение (768×1024 portrait), SHA-256, дубликаты
- Блокировка опасных типов: HTML, JS, SVG, ZIP, EXE, DLL, SH, PY и ещё 8 типов
- Видео-валидация (ffprobe): контейнер, кодек (h264/vp8/vp9/av1), длительность ≤30с, FPS ≤30, аудио запрещено
- GIF-валидация: сигнатура, кадры ≤300, длительность ≤15с
- **Загрузка `.mov` пользователем запрещена**
- Безопасный предпросмотр через backend proxy
- Статусная модель: draft → pending_review → in_review → approved / rejected / validation_failed / archived
- Версионирование: замена файла — новая версия, старая сохраняется

### Модерация креативов

- Очередь модерации: `/creatives/moderation/queue`
- Действия: отправить на проверку, одобрить, отклонить, вернуть на доработку, архивировать
- **Maker-checker обязателен:** создатель не может одобрить свой креатив
- **Журнал аудита обязателен для каждого действия модерации**
- Карточка креатива: предпросмотр, статусы, история, причина возврата
- **Режим `pilot_dev` разрешает ручную модерацию** (audit warning)
- **Имитация проверки безопасности (fake AV pass) запрещена**

### Кампании

- CRUD кампаний: создание, редактирование, статусный жизненный цикл
- **Привязка креатива: только одобренный креатив** (status=approved)
- Gate привязки: неодобренные креативы отклоняются
- Статусы: draft → pending_approval → approved → published

### Расписание и занятость

- CRUD расписаний, слоты по дням недели и времени
- Занятость эфира: `occupancy_pct`, `sold_out`, причины на бизнес-языке
- Инвентарь: прогноз доступного времени, сводка, снапшот
- Страница `/inventory` — Рекламное время

### Согласование

- Запрос согласования кампании
- **Maker-checker обязателен:** инициатор не может согласовать свой запрос
- Принятие решения: одобрить / отклонить с комментарием
- **Журнал аудита обязателен** для каждого действия согласования

### Публикация (backend-only)

- Конвейер жизненного цикла пакета: draft → pending → approved → manifest_generated → published
- Генерация пакета рекламных материалов (JSON manifest)
- Backend-публикация (только уровень системы)
- **Физическая доставка на КСО заблокирована**

### Отчёты

- 4 типа CSV-экспорта: кампании, занятость эфира, конфликты, публикации
- Безопасная проекция (без UUID, secrets, технических полей)
- RLS в экспорте (рекламодатель видит только свои данные)
- Распределение по статусам, конфликты расписания

### Главный экран и готовность

- Сводка платформы (Platform Summary)
- Конвейер рекламы (Advertising Pipeline) — 6 этапов
- Готовность пилота — 5 P0 блокировок, статус 🔴 NO-GO
- Бизнес-действия (Business Next Actions)
- Страница `/readiness` — Device KPI, checklist, acceptance criteria
- Страница `/readiness/business-acceptance` — новая страница бизнес-приёмки

### Безопасность

- RBAC: 8 ролей, 47 разрешений
- RLS: 7 типов scope, query-level enforcement
- JWT-аутентификация (60 минут)
- Server-side sessions (signed httpOnly cookies)
- Двойной аудит: события входа + административные действия
- Никаких JS/CDN/localStorage
- Безопасная проекция в HTML (без raw UUID, secrets, backend URL)

### Администрирование

- Управление пользователями: создание, блокировка, архивация, назначение ролей
- Управление устройствами: регистрация, статус (heartbeat), readiness badge
- AV readiness endpoint: `GET /api/admin/av-readiness`

|---|---|
| **Физическая доставка пакета материалов на КСО** | P0 блокировка B-03 |
| **Фактические показы (Proof of Play / PoP)** | Нет физического PoP flow |
| **Физический сканер E2E** | P0 блокировка B-01 |
| **48ч+ длительная проверка стабильности** | P0 блокировка B-02 |
| **Развёртывание парка устройств** | P0 блокировка B-06 |
| **Синхронизация агента на КСО** | P0 блокировка B-04 |
| **Реальный AV-сканер (ClamAV)** | Не настроен, `scan_status=not_configured` |
| **Производственный режим AV** | Требуется отдельное решение и approval gate |
| **Многоканальность (Android TV, LED, ESL, Mobile)** | Отложено до v2 |
| **AD/SSO/MFA** | Отложено до v1 production |
| **Production-инфраструктура (мониторинг, backup, CI/CD)** | Отложено до v1 production |
| **Бюджетирование кампаний** | Отложено до v2 |
| **XLSX-экспорт** | Только CSV |
| **Визуальные отчёты (графики)** | Без JS невозможно, отложено до v2 |
| **Нагрузочное тестирование** | Не проводилось |

## Физические gates — всё ещё закрыты

Все 5 физических P0 блокировок остаются в статусе ❌:

1. **B-01: Аппаратный сканер E2E** — не выполнен
2. **B-02: Длительная проверка стабильности (48ч+)** — не выполнена
3. **B-03: Физическая доставка пакета материалов на КСО** — не утверждена
4. **B-04: Синхронизация агента на КСО** — не запущена
5. **B-06: Развёртывание парка устройств** — не утверждено

До закрытия всех 5 блокировок **физический пилот остаётся заблокированным**. Статус пилота: 🔴 NO-GO (Запуск заблокирован).

## Следующие шаги после RC0

1. **Закрытие B-01:** Подключить физический сканер → Scanner E2E.
2. **Закрытие B-03:** Утвердить физическую доставку пакета материалов → доставка на одну КСО.
3. **Закрытие B-04:** Запустить синхронизацию агента на физической КСО.
4. **Закрытие B-02:** 48ч+ длительная проверка стабильности с мониторингом.
5. **Решение по AV:** Принять решение о подключении реального AV-сканера и переходе в `production` режим.
   - **Производственный AV требует отдельного решения.**
   - **В производственном режиме публикация без `scan_status=clean` должна блокировать одобрение и публикацию.**
6. **Закрытие B-06:** Утверждение развёртывания парка устройств → GO/NO-GO пилота.
7. **Пилот одного КСО:** Запуск на одном устройстве, мониторинг, сбор фактических показов.
8. **Масштабирование:** Fleet rollout до 100 КСО.
