# KSO Vendor Integration Questions

**Документ для поставщика КСО ПО.** Подготовлен в рамках Шага 24.1 — предварительный опрос перед началом разработки.

**Дата:** 18 июня 2026
**Статус:** Ожидает отправки поставщику

---

## 1. Цель интеграции

Мы хотим получить возможность показывать рекламные материалы на КСО **только в безопасном состоянии** (idle screen), без какого-либо влияния на кассовый сценарий и процесс оплаты.

**Что это даёт:**
- Дополнительный доход для ритейлера от рекламы на КСО
- Показ происходит только когда КСО не используется покупателем
- Никакого влияния на скорость, стабильность и безопасность кассового процесса

---

## 2. Что мы предлагаем

**Lightweight Sidecar Agent** — минимальный фоновый процесс, который:

1. **Синхронизирует рекламные материалы** с облачного backend через HTTPS (outbound-only)
2. **Сохраняет готовые файлы** (изображения, видео) в общую локальную папку
3. **КСО ПО читает файлы** из этой папки и показывает их на idle screen
4. **КСО ПО пишет события показов** (item показан, длительность) в общий JSONL-лог
5. **Агент отправляет статистику** (heartbeat, статус кэша, события показов) обратно в backend

**Ключевые свойства агента:**
- Outbound-only связь (только исходящие соединения к backend)
- Нет входящих портов на КСО
- Не влияет на кассовый интерфейс (не делает overlay, не управляет экраном)
- При ошибке — тихо отключается, не трогая КСО ПО
- Минимальное потребление ресурсов

---

## 3. Что мы НЕ делаем

Наш агент **никогда:**

- Не вмешивается в процесс оплаты
- Не перехватывает и не читает чековые данные
- Не собирает персональные данные покупателей
- Не делает скриншоты экрана КСО
- Не использует камеру или микрофон
- Не открывает входящие сетевые порты
- Не управляет КСО ПО (не запускает/останавливает/меняет настройки)
- Не выполняет удалённые команды с backend
- Не делает overlay поверх критичных экранов (оплата, возврат, ошибка)
- Не хранит и не передаёт данные банковских карт

## 6. Требования информационной безопасности

Наш агент соблюдает следующие правила ИБ. Пожалуйста, проверьте, нет ли конфликтов с вашими политиками безопасности.

### Сетевые

- **Outbound-only:** только исходящие HTTPS-соединения от КСО к нашему backend
- **TLS 1.2+** с валидацией сертификатов (никаких self-signed или `--insecure`)
- **Нет входящих портов** на КСО (только localhost для внутреннего взаимодействия, если понадобится)
- **Certificate pinning** опционально

### Данные

- **Нет персональных данных покупателей** — агент их не собирает и не хранит
- **Нет платёжных данных** — не читает суммы, транзакции, карты
- **Нет чековых данных** — не читает и не перехватывает чеки
- **Нет скриншотов** экрана КСО — агент не делает снимки экрана
- **Нет данных с камеры / микрофона**
- **Нет локальных путей** (`local_path`, `file_path`) в запросах к backend
- **Секреты не попадают в логи** — `device_secret` и JWT маскируются

### Хранение

- `device_secret` (ключ устройства) хранится в отдельном защищённом файле (права 0400)
- JWT-токены — **только в оперативной памяти**, никогда на диске
- Медиафайлы проверяются по SHA256 после скачивания

### Отказоустойчивость

- При любой ошибке агента — **fail_silent**: тихая остановка, без влияния на КСО ПО
- Агент не крашит и не зависает — изолированный процесс
- При недоступности backend — агент продолжает работу в offline-режиме (если разрешён конфигурацией)

### Удалённое управление

- **Нет remote shell / remote command execution** в первой версии
- Агент только запрашивает данные с backend (poll-based). Backend не отправляет команды агенту (нет push)

## 7. Требования к отказоустойчивости

| Ситуация | Поведение агента | Влияние на КСО |
|---|---|---|
| Агент упал (crash) | Перезапускается и продолжает работу | Нулевое. КСО работает как обычно |
| Backend недоступен | Работает по последнему загруженному manifest (offline-режим) | Нулевое. Показ продолжается из кэша |
| КСО выходит из idle | Агент немедленно прекращает обновление медиа-папки | Нулевое. КСО ПО само управляет экраном |
| Ошибка проверки медиафайла (SHA256) | Файл удаляется и перезагружается | Нулевое. Файл просто не показывается |
| Переполнение дискового кэша | Старые файлы удаляются (LRU) | Нулевое. Кэш самоочищается |
| Агент завис (CPU 100%) | Ограничение по CPU из конфигурации | Минимальное. КСО ПО приоритетно |
| Необходимо срочно отключить агент | Администратор останавливает процесс | Нулевое. КСО продолжает работу |

## 8. Требования к тестированию

Для разработки и тестирования нам потребуется:

- **Пилотная КСО** (или тестовый стенд с ПО КСО) для интеграционного тестирования
- **Документация по API состояний КСО** (idle, scanning, payment, error, service mode)
- **Возможность удалённого доступа** к тестовой КСО (SSH / RDP / VNC) — если физический доступ невозможен
- **Тестовый режим КСО ПО**, в котором можно безопасно симулировать:
  - idle → сканирование → оплата → idle
  - ошибку КСО
  - сервисный режим
  - отключение сети
- **Контакт технического специалиста** с вашей стороны для оперативной связи
